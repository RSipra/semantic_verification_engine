# ADR-P2-002: Medallion Data Architecture for Dataset Evolution

**Status:**  ✅ Accepted <br>
**Phase:** Phase 2 (System & Semantic Intelligence)


## Decision
Adopt a Medallion data architecture to organize the trivia dataset into **Bronze (Raw datalake)**, **Silver (validated)**, and **Gold (release-ready)** layers.


## Options considered
1. **Single mutable dataset**. Maintain one dataset that is continuously updated as new questions, corrections, and metadata are added.
2. **Medallion architecture (Bronze → Silver → Gold)**. Separate raw generation, validation, and release preparation into explicit, versioned layers.

## Selected option
Option 2 (Medallion architecture; Bronze → Silver → Gold)

## Rationale
As the content factory evolved beyond simple question generation, a single mutable dataset became difficult to reason about and risky to modify. The Medallion architecture introduces clear staging boundaries:

- **Bronze (Raw):** newly generated or ingested question records, stored as-is for traceability.
- **Silver (Validated):** schema-validated, deduplicated, checked for hallucinations, and QA-approved questions.
- **Gold (Release-Ready):** a stable, versioned dataset that serves as a **handoff boundary** between the Content Factory and the Context Refinery. The Gold dataset includes stable identifiers (e.g., `original_question_id`) and normalization required to convert records back into a baseline-compatible form, but does not yet include downstream semantic or NLP enrichments.

This separation provides safe checkpoints between generation, validation, and downstream feature engineering without prematurely coupling the Content Factory to the Context Refinery.

## Trade-offs
**Pros:**
- Clear separation between generation, validation, and downstream consumption
- Stable interface between the Content Factory and Context Refinery
- Strong traceability via persistent identifiers
- Reduced risk of enrichment logic contaminating validated content

**Cons:**
- Increased operational overhead from maintaining multiple dataset layers
- Additional versioning and bookkeeping complexity across Bronze, Silver, and Gold

## Risks
- Increased structural complexity may slow iteration if discipline is not maintained
- Bypassing layers during experimentation could undermine traceability guarantees
- Dataset version sprawl if older layers are not periodically pruned or archived

## Consequences & Future considerations
- The Gold dataset functions as a buffer and contract to the Context Refinery.
- Enrichment (NER, embeddings, semantic features) is explicitly deferred to the Context Refinery
- This design allows the baseline dataset to be regenerated or transformed consistently using stable identifiers
