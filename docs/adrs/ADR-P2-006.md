# ADR-P2-006: API Framework Selection (FastAPI)

**Status:** ✅ Accepted
**Phase:** Phase 2 (System & Semantic Intelligence)


## Decision
Use FastAPI as the API framework for the runtime service instead of Flask.

## Options considered
1.  **Flask**. A lightweight, synchronous web framework with minimal built-in validation or async support.
2. **FastAPI**. A modern, asynchronous framework with native schema validation and automatic API documentation.

## Selected option
Option 2 (FastAPI)

## Rationale

FastAPI was selected because it enforces a clean, interface-agnostic API boundary and aligns with common ML serving patterns (schema-driven validation, stateless requests, async handling), supporting both the current game runtime and future learning goals.

### Additional benefits

While not primary decision drivers, FastAPI also provides:

- Built-in request and response validation using existing Pydantic schemas
- Automatic OpenAPI / Swagger documentation for easy inspection and testing
- Native support for asynchronous request handling, avoiding blocking during I/O or future inference calls

These benefits reduce the amount of manual validation and request-handling code needed and improve overall clarity.

## Trade-offs
**Pros:**
- Native asynchronous request handling
- Automatic request/response validation using Pydantic
- Built-in, interactive API documentation
- Better alignment with future runtime evolution (e.g., bounded AI inference)

**Cons:**
- Learning curve for async/await patterns
- Slightly higher conceptual overhead compared to minimal Flask apps

## Risks
- Incorrect use of async patterns could introduce subtle bugs if not carefully implemented
- Overengineering risk if advanced features are used prematurely

## Consequences & Future considerations
- The runtime API is future-proofed for asynchronous operations without requiring a framework migration.
- Schema definitions are shared consistently across pipelines and runtime.
- Future runtime enhancements (e.g. SLM-backed interaction handling) can be introduced without revisiting the core API framework choice.
