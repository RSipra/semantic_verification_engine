# ADR-P2-010: Pipeline Orchestration (Prefect)

**Status:** ✅ Accepted  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision
Use Prefect to orchestrate the Phase 2 data pipelines instead of running scripts manually or heavier workflow platforms.

## Options considered
1. **Manual execution**. Pipelines are run as standalone scripts in sequence, with ordering managed manually.
2. **Apache Airflow**. A full-featured workflow orchestration platform using DAGs and operators.
3. **Prefect**. A Python-native orchestration framework designed for lightweight, modular pipelines.

## Selected option
Option 3 (Prefect)

## Rationale
As the system evolved into a multi-step pipeline (question generation → validation → ingestion), manual execution became fragile. Failures were hard to track, retries were manual, and it was difficult to understand which step failed or why.

Airflow was considered but rejected because:
- It introduces significant infrastructure and configuration overhead
- Pipeline logic is expressed through operators and DAG definitions rather than plain Python
- It is better suited for large, long-running, shared production workflows than a focused, evolving project

Prefect was selected because:
- Pipeline logic remains Python, making it faster to adopt and easier to integrate with existing pipeline code.
- It integrates naturally with Pydantic validation and SBERT-based checks
- Retries, logging, and run tracking are provided without heavy infrastructure
- Dependencies between steps are explicit and readable
- Pipeline steps can branch based on validation or deduplication outcomes, making it easier to handle ambiguous cases or introduce human review when needed, aligning with the status/payload pattern introduced earlier (see [ADR-P1-003](ADR-P1-003.md)).

**Pros:**
- Clear dependency management between pipeline stages
- Automatic retries and visibility into failures
- Records logs, task states, parameters, and retries per pipeline run in a single interface, simplifying debugging and inspection without custom logging infrastructure.
- Minimal refactoring of existing Python code

**Cons:**
- Adds a new dependency to the project
- Requires running a local Prefect service during development
- More structure than ad-hoc scripts

## Risks
- Orchestration overhead may be unnecessary if pipelines remain small and infrequent
- Overuse could lead to unnecessary complexity if not kept scoped

## Consequences & Future considerations
- Pipeline execution becomes repeatable, inspectable, and easier to debug
- The system is prepared for future automation without committing to heavy infrastructure
- If pipelines grow in complexity or frequency, Prefect can scale without redesign
- If orchestration proves unnecessary, flows can be simplified back into standalone scripts with limited changes
