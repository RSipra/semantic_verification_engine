# ADR-P2-001: Shift-Left Enrichment Strategy (Hints & Explanations)

**Status:** ✅ Accepted <br>
**Phase:** Phase 2 (System & Semantic Intelligence)


## Decision
Generate hints, explanations, and fun facts during the Phase 2 content factory pipelines (build time), rather than generating them dynamically at runtime in Phase 3.

## Options considered
1.  **Runtime generation via SLM**:. Generate hints, explanations, and fun facts dynamically during gameplay using the Phase 3 SLM.
2. **ETL-time generation (Shift Left)**. Generate and validate hints, explanations, and fun facts offline as part of the Phase 2 dataset enrichment process.


## Selected option
Option 2 – ETL-time generation; shift Left

## Rationale
An early design consideration was to rely on the Phase 3 SLM to generate hints, explanations, and fun facts dynamically at runtime. While feasible, this would increase the cognitive responsibility of the SLM and introduce variability into gameplay responses. By shifting content enrichment left into the Phase 2 pipelines:

- **Stability & Correctness:**  
  Hints and explanations are generated once, validated, and versioned as part of the dataset. This avoids inconsistent or incorrect runtime generation and reduces the risk of answer leakage.
- **SLM Scope Control:**  
  The Phase 3 SLM is limited to well-defined interaction behaviors (e.g., tone, clarification, ambiguity handling) rather than open-ended content generation.
- **Runtime Performance:**  
  Eliminates the need for generative inference during gameplay, keeping runtime latency low and predictable.
- **Operational Simplicity:**  
  Reduces model complexity and avoids placing content-generation logic in the runtime path.

This design treats hints, explanations, and fun facts as **first-class dataset features**, rather than ephemeral runtime outputs.

## Trade-offs
**Pros:**
- More stable and predictable gameplay
- Reduced runtime complexity and inference cost
- Easier validation and quality control
- Clear separation between content generation and interaction logic

**Cons:**
- Content is static once deployed
- Hints cannot adapt dynamically to a specific incorrect user answer

## Risks
- Static hints may feel less adaptive in edge cases
- Changes to hint strategy require regenerating the dataset

## Consequences & Future considerations
- Phase 3 SLM remains lightweight and interaction-focused
- Runtime behavior stays deterministic and cost-controlled
- More dynamic hinting remains a possible future enhancement if user experience data justifies increased runtime complexity
