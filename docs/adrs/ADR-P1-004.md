# ADR-P1-002: Stateful VM Deployment Strategy (Phase 1 MVP)

**Status:** Accepted <br>
**Phase:** Phase 1 (MVP)


## Decision
To deploy the Phase 1 MVP as a **Stateful, Virtual Machine-based** application. The Python game loop will run continuously within a Docker container, using **GoTTY** to translate standard I/O to a WebSocket stream for the browser. This replaces the originally planned Serverless/Stateless architecture for this specific phase.

## Options considered
1.  **Stateless Web App (FastAPI + Cloud Run):** Wrap the game logic in a REST API immediately. Requires refactoring the `View` layer to HTML/JSON and removing blocking `input()` calls.
2.  **Stateful VM (GCE + GoTTY):** Run the existing CLI code on a persistent Virtual Machine, using a Web-to-Shell wrapper to render the interface in a browser.
3.  **Local Execution Only:** Distribute the code via GitHub and require users to run it locally.

## Selected option
**Option 2: Stateful VM (GCE + GoTTY)**

## Rationale
This option aligns best with the strategic goal of Phase 1: maximizing **User Experience (UX)** and **Demonstration Value** with minimal code refactoring.
* **UX Preservation:** The project's "hook" is the immersive terminal interface (using the `rich` library). Option 1 would degrade this to a static form; Option 2 preserves all animations and real-time interaction.
* **Speed to MVP:** We can deploy the existing Python logic immediately without rewriting the core loop for a stateless environment.
* **Systems Thinking:** This choice demonstrates the ability to match infrastructure (Stateful VM) to the application's current lifecycle stage (Interactive Session).

## Trade-offs
**Pros:**
* **High Fidelity:** Browser experience is identical to the local terminal (colors, animations, blocking I/O).
* **Zero Refactoring:** No need to implement complex state management or database sessions yet.
* **Cost Efficiency:** Fits entirely within the Google Cloud "Always Free" tier (`e2-micro`).
* **Persistence:** The VM maintains the game state in memory, preventing timeouts during user "thinking time."

**Cons:**
* **Single Point of Failure:** If the VM crashes or restarts, all active sessions are lost instantly.
* **Scaling Limit:** Restricted to vertical scaling (max 1GB RAM on free tier); cannot auto-scale to handle traffic spikes.
* **Infrastructure Debt:** This entire deployment configuration is temporary and must be discarded for Phase 2.

## Risks
* **Memory Exhaustion:** The `e2-micro` instance has limited RAM (1GB).
    * *Mitigation:* Configure a 2GB Swap file on the VM boot script.
* **Cost Overage:** Misconfiguration could lead to billing for egress or compute.
    * *Mitigation:* Set a strict budget alert at $1.00/month.

## Consequences & Future considerations
* **Phase 2 Pivot:** This decision creates a "hard stop" for Phase 2. We acknowledge that the GoTTY wrapper will be removed and the infrastructure will shift to **Cloud Run (Serverless)**.
* **View Layer Refactor:** We accept that the `rich` console code is technical debt. It will be replaced by **Jinja2/HTMX** templates in the next phase to support the stateless architecture.
