# ADR-P2-014: Answer Shape Pattern Monitoring

**Status:** ✅ Accepted  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision

Monitor answer shape patterns by question type at the dataset level after each Content Factory run, using answer-length distributions to detect drift and preserve consistency with empirically observed Phase 1 patterns. 

Answer shape is guided primarily through prompting. Monitoring is used to surface deviations rather than to enforce hard constraints.

## Options considered

1. **No monitoring**.  
   Rely solely on prompting and assume answer shape remains stable as new data is generated.

2. **Hard enforcement at generation time**.  
   Enforce strict length or format constraints on LLM outputs to guarantee answer shape.

3. **Post-run pattern monitoring**.  
   Track answer-length distributions by question type after each pipeline run and surface drift through the `publishing` pipeline dashboard.

## Selected option
Option 3 (Post-run pattern monitoring)

## Rationale

Phase 1 analysis revealed a stable relationship between question type and answer shape. In particular, Explanatory (EX) answers were consistently longer than short-answer types such as Factual Recall (FR), Yes/No (YN), and Multiple Choice (MCQ), with little to no overlap in their interquartile ranges.

This pattern simplifies answer checking, feedback logic, and gameplay behavior. However, as new questions are generated using LLMs, there is a risk of gradual verbosity drift that could erode this structure over time.

The system does not try to enforce this pattern with hard rules. Instead, answer shape is guided through prompting, and the pattern is monitored at the dataset level. After each Content Factory run, answer-length distributions by question type are computed and shown in the publishing dashboard so drift is easy to spot.

Monitoring is admin-facing and manual through the `publishing` pipeline dashboard.

(*Dashboard note*: answer-length distributions by question type are shown with an indication if the interquartile range of Explanatory answers overlaps with other question types.)

## Trade-offs

**Pros:**
- Preserves useful dataset-level patterns without rigid enforcement  
- Keeps the system flexible and adaptable to edge cases  
- Aligns with offline-first, batch-oriented design  
- Enables early detection of drift before it affects downstream logic  

**Cons:**
- Requires manual inspection when drift is detected  
- Does not prevent individual outliers at generation time  

## Risks
- Subtle drift may go unnoticed without regular dashboard review  
- Patterns may evolve naturally as question complexity increases and require reinterpretation  

## Consequences & future considerations

- Answer-length distributions are recomputed on each pipeline run and included in the `publishing` pipeline dashboard  
- Visual signals (e.g. box plots, IQR overlap indicators) guide prompt refinement and data inspection  
- If drift becomes persistent, additional safeguards can be introduced incrementally  
- This pattern establishes a general approach for monitoring other dataset-level properties over time.
