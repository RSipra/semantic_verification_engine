# ADR-P2-009: Data Versioning (DVC)

**Status:** ✅ Accepted  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision

Use DVC (Data Version Control) with a Google Cloud Storage (GCS) backend to version datasets across the Bronze, Silver, Gold, and Production stages.

## Options considered
1. **No explicit data versioning**.  Rely on local files or ad-hoc naming conventions.
2. **Manual versioning**. Store datasets with timestamped filenames or folders.
3. **DVC-based data versioning**. Track datasets alongside code with a remote object store backend.

## Selected option
Option 3 (DVC-based data versioning)

## Rationale
Since the content factory uses the medallion pattern (Bronze → Silver → Gold → Production), the datasets are expected to evolve as questions are regenerated, validation rules are refined, and enrichment logic is updated. These changes will not always align directly with code changes.

Without data versioning, it would become difficult to trace the datasets back to specific pipeline runs; to identify root-causes for data issues (e.g. prompt, generation logic). And although regeneration is expected to be inexpensive and bounded, relying solely on re-running generation would obscure what changed and why.

DVC is proposed as a lightweight solution because it is only used with well-defined batch runs. Dataset generation is expected to be infrequent and to taper off as the dataset stabilizes. So DVC adds traceability with minimal operational overhead while keeping the workflow simple and contained. Data artifacts will be stored in GCS, with version references tracked in the repository alongside code and configuration. This makes changes to Bronze, Silver, Gold, and Production datasets explicit and easier to reason about as the system grows.

## Trade-offs

**Pros:**
- Enables reproducible pipeline runs across dataset versions
- Tracks data changes alongside code changes
- Scales to large datasets without bloating the Git repository
- Integrates naturally with the existing cloud storage setup

**Cons:**
- Adds tooling and workflow overhead
- Requires discipline to keep data and code versions aligned

## Risks

- Misuse or partial adoption could lead to inconsistent dataset states
- Added complexity for a solo project if not kept lightweight

## Consequences & Future considerations

- Dataset evolution becomes explicit
- Pipeline outputs at each medallion stage can be traced and reproduced
- Supports future debugging, experimentation, and comparison across runs
- If the system remains small and static, DVC usage may be simplified; if it grows, DVC provides a foundation for more advanced data lineage and experiment tracking
