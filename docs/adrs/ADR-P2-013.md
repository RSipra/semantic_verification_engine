# ADR-P2-013: Centralized Semantic Validation Strategy

**Status:** ✅ Accepted  
**Phase:** Phase 2 (System & Semantic Intelligence)<br>
**Updated [10 June 2026]**: semantic authority relocated from Gold to Silver to separate the system-of-record (Silver) from the runtime contract (Gold). Core centralization decision unchanged.

## Decision

Adopt a centralized semantic validation strategy based on **precomputed SBERT embeddings**, reused across offline pipelines (including deduplication, ambiguity detection, and answer validation), rather than recomputing semantic representations independently for each semantic check. Semantic similarity is treated as a shared system capability rather than a pipeline-specific optimization.


## Options considered

1. **On-the-fly semantic computation per use case**  
   Each pipeline (e.g. deduplication, answer validation) computes SBERT embeddings independently as needed.

2. **Centralized, precomputed semantic representations**  
   Maintain versioned semantic representations of the Silver dataset that are reused consistently across semantic validation tasks.

## Selected option

Option 2 (centralized, precomputed semantic representations)

## Rationale

During implementation exploration, it became clear that semantic similarity (via SBERT) is not a one-off optimization for deduplication but a recurring operation across the system. Treating each usage independently would introduce duplicated logic, inconsistent similarity thresholds, and unnecessary model inference.

The Silver dataset evolves through bounded, batch-oriented updates, making it well-suited to precomputation. Semantic representations can therefore be generated once per dataset version and reused across all semantic validation steps.

This decision keeps semantic validation:
- **Offline**, rather than part of the runtime critical path  
- **Efficient**, by avoiding repeated embedding inference  
- **Consistent**, by comparing against a stable semantic reference per dataset version  

The specific indexing or retrieval mechanism (storage) is intentionally left as an implementation detail.

### Semantic invariants & ownership

The following invariants govern how semantic representations are produced, stored, and consumed across the system:

1. **Silver dataset (audit ledger) is the semantic authority**  
   - Embeddings are versioned and stored in Silver; Gold carries a derived runtime projection.    
   - No other pipeline mutates or regenerates embeddings.

2. **`qa_validation` is the sole embedding producer**  
   - All SBERT embeddings (for both questions and answers) are computed only within the `qa_validation` pipeline.  
   - Other components consume semantic representations but do not perform embedding computation themselves.

3. **Semantic deduplication rule**
   - New questions are checked for semantic duplication only against Silver embeddings.
   - Although the production dataset also contains embeddings, it is derived from the Silver dataset and may include additional features.
   - Using Silver-owned embeddings ensures deduplication is based on the single source of truth rather than on derived or augmented data.

4. **SBERT models are stateless artifacts** 
   - The SBERT model has no internal state.
   - A specific SBERT model version is copied into the runtime container when the container is built.
   - Offline pipelines use the same SBERT model version.
   - If the SBERT model changes, the runtime container must be rebuilt and semantic embeddings must be regenerated

5. **Single SBERT version enforced system-wide**  
   - One SBERT model version is enforced across both offline processing and runtime inference.  
   - Model upgrades are explicit, versioned events.

## Trade-offs
**Pros:**
- Eliminates duplicated semantic logic across pipelines  
- Reduces repeated model inference and computation  
- Ensures consistent semantic behavior system-wide  
- Aligns with batch-oriented pipeline design  
- Keeps runtime latency isolated from semantic complexity  

**Cons:**
- Requires managing semantic representations alongside dataset versions  
- Introduces centralized coupling: changes to embeddings or models affect multiple pipelines  

## Risks
- If semantic representations fall out of sync with Silver dataset versions, validation accuracy could degrade  
- Early assumptions about embedding models or similarity thresholds may require revision as usage expands  

## Consequences & future considerations

- Semantic validation becomes a first-class, reusable system capability    
- New semantic checks reuse existing embeddings, avoiding redundant computation and duplicated logic. 
- Embedding storage or lookup mechanisms may change over time without altering the definition or ownership of semantic validation.
- The sentence embedding model used for semantic comparison (currently SBERT) may be replaced in the future through an explicit, versioned change that regenerates Silver embeddings and rebuilds runtime artifacts.
- *Open decision: SBERT model source (fine-tuned vs. off-the-shelf)*:
The system currently uses an off-the-shelf SBERT model. Fine-tuning remains an open, deferred decision rather than a committed path. Current direction is to not fine-tune, because the ambiguous cases fine-tuning would target (long, multi-claim Explanatory answers) are better addressed by the LLM/SLM judge tier and the planned claim-decomposition step. Fine-tuning would optimize the SBERT layer for a problem the judge tier already handles more effectively. Revisit if the judge tier proves insufficient or if its cost/latency motivates pushing more resolution into local inference.
