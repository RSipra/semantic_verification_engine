# ADR-P2-003: Adapting offline (feature) store pattern to Context Refinery

**Status:** ✅ Accepted <br>
**Phase:** Phase 2 (System & Semantic Intelligence)


## Decision
Consolidate all derived-feature generation into the Context Refinery, making it the single owner of every feature derived from the core generated columns, both descriptive features (e.g. question/answer length, answer type, question type, interrogative keywords) and contextual/semantic features (e.g. NER tags, difficulty labels, thematic categories). Descriptive-feature generation is moved out of the `qa_validation` pipeline.

The `qa_validation` pipeline / Content Factory retains only generation and validation responsibilities.

## Options considered
1. **Split feature ownership (original arrangement).** Descriptive features generated in `qa_validation`; contextual/semantic features generated in the Context Refinery. Ownership of "derived features" is split across two subsystems.
2. **Consolidate all derived features in the Context Refinery.** `qa_validation` owns generation and validation only; the Context Refinery owns all feature enrichment derived from the validated core columns.

## Selected option
Option 2 (consolidate all derived-feature ownership in the Context Refinery).

## Rationale
Originally, feature engineering was spread across two subsystems (descriptive features in `qa_validation`, contextual features in Context Refinery). This blurred responsibilities.

Comparing the design against the offline feature-store pattern clarified that the Context Refinery should own *all* feature engineering. Consolidating there gives a clean separation of concerns: 
- `qa_validation` is responsible for producing and validating the core 
content (the system-of-record columns), 
- the Context Refinery is responsible for everything *derived* from those columns. Derived features now have a single owner and a single place to 
evolve.

This adapts the *offline portion* of the feature-store pattern, i.e. centralized, decoupled, offline enrichment, but stops short of a full feature store (no serving layer, registry, or online/offline parity), which would be premature at batch, demo scale.

## Trade-offs
**Pros:**
- Single owner for all derived features; clear separation between validation (qa_validation) and enrichment (Context Refinery)
- qa_validation simplified to one concern (generate + validate core columns)
- Reduces duplication and ambiguity about where feature logic lives

**Cons:**
- Adds a dependency. The Production dataset is only complete after both validation and enrichment have run, so the two stages must stay coordinated
- the descriptive features are cheap to compute and were conveniently colocated with validation; moving them out trades that convenience for cleaner ownership;

## Risks
- Coordination across the validation→enrichment boundary. Because core columns (`qa_validation`) and derived features (Context Refinery) are now produced in separate stages, a schema change to the core columns must be propagated to the enrichment logic, or features could be computed against stale assumptions
- No feature registry. Feature definitions are tracked by schema and convention. Acceptable at current scale, but as feature count grows, the absence of a registry could make it harder to track lineage and dependencies

## Consequences & Future considerations
Descriptive-feature generation is removed from qa_validation and relocated to the Context Refinery; qa_validation no longer produces derived features, only validated core columns.
