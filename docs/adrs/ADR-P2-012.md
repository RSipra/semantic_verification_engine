# ADR-P2-012: Dataset Enrichment Pipeline (Notebook → Script Transition)

**Status:**  🚧 Proposed  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision
Adopt a **Bootstrap & Shed** approach for dataset enrichment: prototype transformation logic in a notebook for initial legacy migration and experimentation, then refactor and retain only the core logic as a streamlined, script-based enrichment pipeline.

## Options considered
1. **One-off migration script**. Write a temporary script for legacy data migration, discard it, and build separate tooling later for enrichment.
2. **Fully generalized enrichment pipeline upfront**. Design and implement a full pipeline capable of handling both legacy and future schemas from the start.
3. **Bootstrap & Shed (Notebook → Script)**. Prototype enrichment logic in a notebook, then extract and harden only the reusable transformation logic into a permanent pipeline.

## Selected option
Option 3 (Bootstrap & Shed)

## Rationale
The logic required to migrate legacy data (cleaning, scoring, validation, feature injection) largely overlaps with the logic needed for future enrichment and patching of the Gold dataset.

Using a notebook first enables rapid iteration and human-in-the-loop verification during this one-time migration and early experimentation phase. Planning explicitly to *shed* the notebook and legacy-specific code afterward avoids long-term technical debt, while preserving the validated core logic for ongoing use.

This allows for more balance between development speed and long-term maintainability.

### Gold bootstrap requirement

The manually curated legacy dataset must be used to bootstrap the initial (first-pass) Gold dataset using the qa_validation pipeline before any synthetic questions are added.

This ensures that legacy questions become part of the canonical reference set used for semantic deduplication. Without this bootstrap step, legacy questions would be incorrectly treated as duplicates and dropped when compared against later synthetic generations.

The bootstrap is a one-time operation. After Gold is initialized, all subsequent synthetic questions are validated and deduplicated against the existing Gold dataset using the same pipeline logic.

## Trade-offs
**Pros:**
- Avoids duplicating transformation logic for legacy vs. future data
- Enables rapid iteration during early, uncertain phases
- Keeps the final enrichment pipeline focused and lightweight
- Maximizes reuse of validated logic

**Cons:**
- Early enrichment runs rely on manual notebook execution
- Temporary reliance on notebooks increases risk of execution-order errors

## Risks
- **Overwrite / lineage confusion**: Enrichment reads from and writes to Gold datasets  
  *Mitigation*: Enforce strict versioned outputs; never mutate datasets in place.
- **Loss of legacy reprocessing capability**: Deleting migration-specific code may limit future re-ingestion  
  *Mitigation*: Archive raw data and the migration notebook in version control.
- **Notebook execution drift**: Code may depend on implicit state or execution order  
  *Mitigation*: Require a “Restart & Run All” check before any notebook is considered valid.

## Consequences & Future considerations
- Phase 2 favors agility and correctness over full automation
- Core transformation logic should be written as Python functions to ease refactoring
- The enrichment pipeline can later be automated via Prefect once the logic stabilizes
- This pattern establishes a repeatable approach for future feature additions and dataset patches
