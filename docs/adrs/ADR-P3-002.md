# ADR-P3-002: Single GameController-Managed SLM

**Status:** ✅ Accepted<br>
**Phase:** Phase 3 (Model assissted game master)


## Decision
Use a single instruction-tuned Small Language Model (SLM) to handle multiple tasks (e.g., grading, persona response, redirection) instead of deploying multiple specialized agents or task-specific models.

## Options considered

1. **Task-specific specialization using LoRA adapters**. Separate adapters per task (e.g., grading, persona), loaded or swapped at runtime.

2. **Single unified model**
One instruction-tuned SLM, with task switching controlled via structured system prompts.
 

## Selected option
Option 2 {Single unified instruction-tuned model}

## Rationale
Both approaches were considered from the outset. A single instruction-tuned model was selected as the initial implementation to minimize runtime complexity while the task surface remains small.

Using one model avoids loading multiple sets of weights into memory and simplifies edge deployment. Task switching is handled through explicit system prompts (e.g., [TASK: GRADE], [TASK: PERSONA]), allowing fast mode changes without additional infrastructure. At the demo stage, a single instruction-tuned model provides the best balance between capability, runtime simplicity, and cost for Phase 3. 

## Trade-offs
**Pros:**

- Lower memory footprint compared to running multiple models or adapters.
- Simpler runtime architecture (one model, one inference path).
- Faster task switching using prompt-level control.
- Easier to reason about failure modes and behavior consistency.

**Cons:**
- Requires careful instruction tuning to avoid task interference.
- Less modular than fully separate agents.
- Task-specific optimizations are constrained by a shared model capacity.

## Risks
- Instruction bleed-through. The model may confuse tasks if prompts are poorly scoped.
- Capacity limits: A single small model may struggle if too many responsibilities are added.
- Debugging complexity: Errors may be harder to attribute to a specific “mode” than with separate agents.

Mitigation includes:

- Strict separation of control and data. The Controller constructs the system prompt; user input (answer) is sanitized and injected strictly as a data payload, preventing direct instruction override.
- Explicit task separation during training.
- Keeping the task set intentionally small in Phase 3.


## Consequences & Future considerations
- Phase 3 runtime remains lightweight and suitable for edge deployment.
- Knowledge distillation workflows can target a single model rather than multiple agents.
- If task complexity grows or interference becomes problematic, the architecture can evolve toward:
    - multiple LoRA adapters, or
    - specialized agents introduced selectively
