# ADR-P2-022: GLiNER for Dynamic Thematic Tagging & Entity Extraction

**Status:**  proposed / backlog
**Phase:** Phase 2 (main development)

## Decision
In full development of Phase 2, adopt GLiNER (Generalist Model for Named Entity Recognition) to dynamically categorize questions, extract entities, and append thematic tags to the dataset without relying on static lists or fine-tuned custom models.

## Options considered
1. **GLiNER (zero-shot NER)**: Using a pre-trained generalist model to dynamically extract arbitrary, custom entity types (e.g., "Spell", "Potion", "Magical Creature") via prompting.
2. **NER gazetteers**: Maintaining static dictionaries of character, spell, and location in Harry Potter to match against question text (defined, bounded scope = series books)
3. **Fine-tuned custom NER models**: Training a traditional model (like spaCy or BERT) using thousands of manually annotated examples to recognize specific lore categories.

## Selected option
Option 1 (GLiNER)

## Rationale
At scale static Gazetteers can become unmaintainable and brittle when facing newly generated or obscure entities. Fine-tuning a custom model requires prohibitive manual annotation. GLiNER solves this by allowing zero-shot extraction of domain-specific entities right out of the box, handling the vast and varied lore of the source material.

## Trade-offs
**Pros:**
- **Zero training data**: Requires no manual annotation or fine-tuning.
- **High flexibility**: Can instantly adapt to new entity classes (e.g. adding "Ministry Department") just by updating the target labels.
- **Automated enrichment**: Enables the Game runtime to support *themed rounds* and algorithmically weight question *difficulty* based on entity rarity.
- **Zero runtime latency**: Because entity extraction is performed as an offline batch process in the Context Refinery, it adds no latency to the live game engine.
**Cons:**
- **Implementation learning curve**: slightly steeper initial curve than writing a simple Python string-matching script needed for gazetteers (but avoids manual curation / building of HP terms dictionary) -> API similar to SBERT but interpreting and tuning entity label design for a HP domain (e.g. *spell* vs *dark magic* vs *curse*) will require iteration.

## Risks
- **Integration time-box**: The unfamiliarity with GLiNER's API could cause development delays during Phase 2 ingestion pipeline setup.
- **Zero-shot accuracy drift**: GLiNER may occasionally miscategorize highly obscure, overlapping entities, requiring occasional human audits of the Gold dataset to ensure gameplay themes remain accurate. If thematic drift is detected during periodic audits, the pipeline should transition to a Hybrid Cascade Model:
    - Pass 1 (Gazetteer): Immediate lookup for known "High-Certainty" entities.
    - Pass 2 (GLiNER): Triggered if Pass 1 fails OR to validate Pass 1 results.
    - Pass 3 (Rationalizer): An automated logic gate that compares both outputs -> Match: Confirm label / Conflict: Flag for human review or default to the broader GLiNER category to ensure game-play variety.

## Consequences & Future considerations
The Context Refinery will have an additional NER *enrichment layer* utilizing GLiNER. Output columns (e.g. `ner_entities`, `ner_themes`) will be features of the Production tier only and do not modify the Gold dataset. Gold remains the stable, validated handoff contract between the Content Factory and the Context Refinery subsystems.
