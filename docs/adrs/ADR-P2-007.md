# ADR-P2-007: Client Architecture (Remote Terminal)

**Status:** ✅ Accepted
**Phase:** Phase 2 (System & Semantic Intelligence)


## Decision
Use a thin client / remote terminal architecture where the backend renders the game interface and the browser acts as a simple display, communicating strictly via HTTP/JSON.

## Options considered
1. **Browser-based UI (HTML / React SPA)**. Build a client-side interface responsible for rendering, interaction flow, and presentation logic.
2. **Remote Terminal (Server-Side Rendering)**. Render the game interface on the server using the existing Rich-based CLI and stream the output to the browser, which acts as a dumb terminal.

## Selected option
Option 2 (Remote Terminal)

## Rationale
The primary driver for this decision is minimizing rework of an already functional CLI implementation that is sufficient for a demo. The Rich-based CLI already delivers an engaging and expressive interaction style, and preserving it avoided re-implementing presentation logic in HTML without improving gameplay or architectural learning outcomes. With this approach:
- The Python backend owns both game logic and visual rendering, using the existing rich-based CLI interface.
- The browser acts as a dumb terminal, receiving rendered output from the server and sending back user input.
- Communication between client and server happens strictly over HTTP, keeping the interface boundary clean.

Building and maintaining a frontend UI would have shifted focus away from the project’s core learning goals in data science and MLOps, so it was intentionally deferred. Additionally, the text-based, retro-style interface may appeal to some users and reinforces the playful, nostalgic tone of the game without requiring additional UI complexity.

## Trade-offs
**Pros:**
- Rapid deployment using existing Phase 1 logic and themes
- No frontend framework or client-side game logic required
- Clear separation between game logic and transport layer (HTTP)
- Backend retains full control over layout, styling, and interaction flow

**Cons:**
- Interface is text-based, limiting visual richness
- Server controls presentation, making it harder to support multiple UI styles
- Adding a full graphical UI later will require a new client implementation

## Risks
- Some users may prefer richer visuals from a browser-based game for a full HP immersive experience.
- Long-term UI flexibility is limited by server-side rendering choices

## Consequences & Future considerations
- Phase 2 focuses on correctness, structure, and deployability rather than visual polish
- The remote terminal serves as a bridge architecture, enabling cloud deployment without frontend rework
- Once the core game logic and runtime behavior are stable, a richer frontend (e.g., a graphical web UI) can be introduced as a feature upgrade. 
- The current stateless, API-driven architecture supports adding new clients without modifying the core game engine.
