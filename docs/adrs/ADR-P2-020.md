# ADR-P2-020: Contextual Integrity Checks of Synthetic Questions (`qa_validation` pipeline) 

**Status:**  Proposed<br>
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision
The `qa_validation` pipeline will utilize a two-tiered architectural approach for contextual integrity validation (grounding, hallucination checks, etc.):

1. **Period rigorous testing**: Following system changes (e.g., generation model swaps, significant prompt modifications), a small, representative sample of generated questions is rigorously tested by a high-reasoning LLM Judge utilizing a modified RAG-Triad framework. This is followed by a manual review of the judge performance.
2. **Bulk contextual integrity checks**  Mass evaluation of the generated dataset utilizing automated circuit breakers to monitor stability and route data:
    - > 95% matches: Stable generation; auto-promote dataset to Silver.
    - 85% - 95% matches: Warning state; pause pipeline and flag batch for Human-in-the-Loop (HITL) review.
    - < 85% matches: Circuit breaker tripped; halt pipeline, drop batch, investigate root cause.

To execute this validation effectively without losing data quality, the pipeline will use categorical error codes along with boolean flags when the LLM Judge assesses individual columns.

## Options considered
1. **SBERT matching quotes:** Rigid string matching. Rejected because it fails creative, spliced quotes.
2. **LLM Judge (binary Pass/Fail)**: Strict true/false grading. Rejected as it aggressively prunes complex questions, leading to quality drift.
3. **LLM Judge (with category grading)**: Selected. Allows nuanced grading and composite score routing. Along with flags, it assigns specific codes for common observed behaviors (e.g. source quote categories: exact match, grounded but spliced quote, fabricated).

## Selected option
Option 3 (LLM Judge with cateogry grading)

## Rationale
- Changes in the generation process (e.g. LLM model swap, model removed, prompt changes) can significantly impact the quality of questions and need to be screened before they can be upgraded to Silver level.
- the methodology uses RAG-Triad for assessing the suitability of a record: 
    - **Source quote verification** source_quote is the source of truth for the question. Does it match the provided grounding text?
    - **Answer correctness** is the answer correctly anchored in the source quote only and does not rely on external Harry Potter lore knowledge.
    - **Question suitability** is the question setup so that it illicits the answer correctly with its contextual framing?
- Options 1-3 were explored and tested in the [phase 2 tracer development](../../notebooks/02_research/07_qa_validation_v0.ipynb).
- when looking at the source_quote grounding:
    - the tracer dataset proved the generating model sometimes splices non-adjacent sentences to drop irrelevant narrative text or dialogue. This tension between Generator creativity and Judge strictness is expected and correct.
    - Strict binary filtering fails these valid questions, risking a drift toward overly simplistic, dry trivia.
    - LLM Judge assigns 0 (exact), 1(partial/spliced), 2 (fabricated) error codes.
- **Cumulative post-processing scoring**: using the judge response, every question begins with a base score (e.g., 100). Each code category for each triad item (source quote, answer, question) will have a deduction associated with it. The total deductions will produce a final score and can change the LLM designated flag.

## Trade-offs
**Pros:**
- Preserves narrative nuance and variety in the dataset.
- Tightly screens generator behavior to protect Gold dataset integrity.
- Automated circuit breakers prevent bulk pipeline failures or database pollution.

**Cons:**
- Relying on a high-reasoning LLM Judge incurs higher API costs than a deterministic local proxy.
- Requires slightly more complex post-processing code to calculate composite scores and route flags.

## Risks
- Judge hallucination/inconsistency: The evaluating LLM might inconsistently apply the categorical codes (e.g. misclassifying a Code 1 acceptable splice as a Code 2 fatal fabrication).
- Threshold calibration: The automated circuit breaker thresholds (95%, 85%) are assumed and will require empirical tuning during the first mass-generation runs to prevent false alarms.

## Consequences & Future considerations
- **Bulk Validation Method (Open Decision):** The decision to perform mass validation via a 100% LLM Judge pipeline versus building a sentence-level SBERT proxy is parked. It can be decided later based on compute costs and time limitations at scale.
- **Pan-Series questions**: Future game updates will require synthesizing questions across multiple books. The categorical LLM Judge foundation built here is flexible enough to handle this by eventually introducing a `question_scope` tag (e.g. 'local' vs. 'synthesis') to bypass adjacency / splicing penalties.
- **SBERT Proxy optimization**: In the bulk validation pipeline, the localized SBERT proxy replaces the prototype  sliding-window text search with a tensor-based approach. Entire chapters are pre-embedded into dense vector matrices and cached in memory. Validation queries are embedded once and evaluated using vectorized matrix multiplication to compute similarity scores across all sentences simultaneously. Sentence-level similarity gradients are then analyzed to distinguish legitimate narrative continuity from unsupported or fabricated content.
