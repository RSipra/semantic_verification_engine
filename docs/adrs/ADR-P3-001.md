# ADR-P3-001: Local SLM Runtime & Offline Knowledge Distillation

**Status:**  ⏳ Deferred [10 June 26]; Previously accepted; reopened as one option within the Phase 3 open design space. Deferred pending scale/requirements. <br>
**Phase:** Phase 3 (Optional Runtime Enhancements)

## Decision
Introduce a local, fine-tuned Small Language Model (SLM) into the runtime to support interaction-level intelligence (persona, feedback, and ambiguity resolution), and train this SLM using offline knowledge distillation from the existing Content Factory → Production dataset setup.

The SLM is embedded directly into the deployment container and operates without external API calls.

## Options considered
1.  **External LLM API calls at runtime (contingency)**. Use a third-party foundational LLM via runtime API calls for persona, feedback, and ambiguity handling. Calls would be mediated through a simple interaction manager to strictly limit call frequency and scope. This option remains available as a fallback if local inference quality proves insufficient or if rapid iteration is temporarily prioritized over cost predictability and air-gapped deployment.
2. **Local SLM fine-tuned offline from the Production dataset**. Train an SLM using the Production dataset shaped by an LLM teacher and deploy it locally as a static runtime artifact.

## Selected option
Option 2 — Local SLM fine-tuned via offline knowledge distillation

## Rationale
**Cost and deployment constraints:**Runtime API calls introduce unbounded operational cost and violate tight free-tier constraints. Local inference provides predictable, capped cost.

**Air-gapped execution:** Embedding the SLM locally preserves the air-gapped deployment model, eliminating runtime network dependencies and external credentials.
**Consistency and control:** Fine-tuning the SLM using the existing Production dataset ensures consistent tone, domain grounding, and answer judgment aligned with the game’s content.
**Reuse of existing infrastructure:** The Content Factory already produces high-quality, validated data using an LLM teacher. Leveraging this dataset for LLM → SLM distillation avoids introducing parallel training pipelines.
**Clear responsibility boundaries:** The SLM augments interaction and ambiguity handling without replacing deterministic logic (dataset truth or SBERT-based semantic checks).

## Trade-offs
**Pros:**
- Predictable runtime cost and latency
- Fully offline, portable deployment
- Consistent persona and judgment aligned with content
- No dependency on external APIs at runtime
- Clear separation between deterministic correctness and learned behavior

**Cons:**
- Larger container image size
- Requires offline fine-tuning and artifact management
- Less flexible than live API-based models for rapid behavior changes

## Risks
- Overfitting to dataset tone or structure if the Production dataset is narrow or biased
- Model staleness if content evolves significantly without retraining
- Debug complexity compared to rule-based interaction logic

## Consequences & Future considerations
- The system remains fully functional without the SLM; Phase 3 is an optional enhancement.
- Because the content domain is bounded, SLM retraining is triggered only by substantive dataset changes and may stop entirely once the dataset stabilizes
- If scale or reuse demands grow, the SLM could be separated into a dedicated service; this is intentionally deferred.
- The same distillation approach could later be extended to other bounded behaviors without altering runtime architecture.
- Contingency path: External LLM API calls remain a viable fallback for experimentation, rapid iteration, or recovery scenarios, but are intentionally excluded from the default runtime path to preserve cost predictability and air-gapped deployment.
