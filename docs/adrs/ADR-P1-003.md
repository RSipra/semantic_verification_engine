# ADR-P1-003: Data ingestion control flow (Status/Payload Pattern)

**Status:**  ✅ Accepted (Retroactive)<br>
**Phase:** Phase 1 (Discovery and Foundation)


## Decision
Adopt a standardized **Status/Payload Tuple** return pattern `Tuple[str, Any]` for the data ingestion orchestration routine:
`src/ds_utils/eda_scripts.py::run_data_ingestion_pipeline`.

## Options considered
1.  **Exception-Based Flow:** Raise standard Python exceptions (e.g., `ValueError`, `RuntimeError`) immediately upon validation failure.
2.  **Variable Unpacking:** Return individual objects for every artifact (e.g., `return main_df, nan_df, dup_report`).
3.  **Status/Payload Pattern:** Return a strict two-item tuple: `(Status_Flag, Data_Artifact)`.

## Selected option
Option 3 (Status/Payload pattern)

## Rationale
The Phase 1 ingestion orchestrator (`run_data_ingestion_pipeline`) is not binary (success vs. crash); it has multiple *soft failure* states that require user intervention. Option 3 was selected because it solves the data transport problem that Options 1 and 2 fail to address.

**1. Comparison vs. Exception-Based Flow (Option 1):**
Standard Python Exceptions are designed for code errors, not data quality reports.
* **The Problem:** If the script finds duplicates, we need to return the specific `duplicates_user_report` (DataFrame) so the user can fix them. Attaching a DataFrame to a `ValueError` is difficult and uncommon.
* **The Solution:** The Status/Payload pattern treats the *error Evidence* (the Payload) as a first-class citizen, allowing the script to pass rich DataFrames back to the user even during a failure state (e.g. possible semantic duplicates for review).

**2. Comparison vs. Variable Unpacking (Option 2):**
Returning multiple objects (`return df, nans, duplicates`) creates a brittle function signature.
* **The Problem:** Early failures would require returning `None` for all subsequent variables (e.g., `return None, None, None`), which clutters the calling code with checks. If we add a 4th artifact later, we break every function call in the codebase.
* **The Solution:** The Status/Payload pattern provides a stable interface. The return signature is always `Tuple[str, Dict]`, regardless of how many items are inside the dictionary payload.


## Trade-offs
**Pros:** Deterministic interface; allows the orchestrator to finish gracefully and report *partial success* states; easy to test.

**Cons:** Non-Pythonic (deviates from standard `try/except` blocks)

## Consequences & Future considerations

**Interactivity**: Enables the Phase 1 orchestrator to function as an interactive tool that returns actionable data rather than just crashing.

**Testability**: This explicit return style will persist in the Phase 2 Prefect migration. It simplifies unit testing by eliminating the need to mock complex exception chains.
