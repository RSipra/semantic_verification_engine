# ADR-P1-002: Dataset storage (CSV vs Parquet)

**Status:** ✅ Accepted (Retroactive) <br>
**Phase:** Phase 1 (Discovery and Foundation)


## Decision
Standardize on **Apache Parquet** for all intermediate and final dataset storage, enforcing schemas via **TypedDict** where applicable.

## Options considered
1.  **CSV (Comma Separated Values):** The universal standard. Human-readable and common, but strictly text-based.
2.  **Parquet (Columnar Binary):** A binary format optimized for analytics that preserves metadata and schema information.

## Selected option
Option 2 (Parquet)

## Rationale
The primary driver is **Data Integrity across Notebooks** and **Schema Preservation**. Parquet preserves the exact Pandas data types (Categoricals, Int64, etc.), ensuring that the dataframe stays intact. Notable issues:

1. **Complex Types:** The trivia dataset contains columns with complex types (e.g., `question_tokens` is a `list[str]`). Storing this in CSV converts the list to a flat string (`"['A', 'B']"`). This forces downstream code to perform fragile string parsing (`literal_eval`) to recover the list, introducing a major chance for silent type-conversion bugs.

2. **The "NumPy vs. Native" schema bug:** When loading data from CSV, Pandas infers numeric types as `numpy.int64`. This causes standard schema checks (e.g., `isinstance(val, int)`) to **fail** because Python treats NumPy integers as different objects than native integers. *Impact:* Valid new questions were being rejected by the validation logic.

3. **Integer Precision & null safety:**: In CSVs, a single missing value forces an entire Integer column to cast to Float (e.g., `ID: 5` becomes `5.0`). This breaks ID-based validation logic (e.g., `if id > 0`).

## Trade-offs
**Pros:**
- **No Parsing Logic:** Eliminates the need to manually recast columns after loading.
- **Performance:** Significantly faster read/write operations and smaller file sizes (compression) compared to CSV.
- **Forward Compatibility:** aligns with the Medallion Architecture (Silver/Gold layers) used in Phase 2.

**Cons:**
- **Not Human Readable:** Cannot be opened in a standard text editor for quick inspection; requires a dataframe viewer or plugin.
- **Dependencies:** Requires `pyarrow` or `fastparquet` libraries.

## Consequences / Risks
**Immediate:** 
1. All notebooks must include `pyarrow` in their environment. 
2. The game needs to read in a Parquet file instead of csv for consistency.

**Long-term:** Sets the standard for the Phase 2 pipeline. The Bronze $\rightarrow$ Silver $\rightarrow$ Gold flow will continue to use Parquet, ensuring type safety throughout the entire lineage.
