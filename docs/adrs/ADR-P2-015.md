# ADR-P2-015: Statistical Quality Control (SQC) for Synthetic Data

**Status:** 🚧 Proposed  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision

Introduce a statistical quality control approach for synthetic question generation using **sampling-based review at the batch level**. This acts as an offline safety net to detect incorrect or hallucinated questions early without requiring full manual inspection of all generated data.

At demo scale, quality control is performed through sequential acceptance sampling, where a small, manually reviewed sample is inspected after each batch run and deeper review is performed only if issues are found. Thresholds and protocols are intentionally exploratory and will be refined through experimentation. Automation is intentionally deferred until empirical error rates and generation frequency justify it.

## Context
The system uses a layered defensive approach to manage data quality. The Content Factory uses LLMs to generate batches of trivia questions. Prompting and grounding reduce the likelihood of incorrect content, but they do not provide guarantees. As identified in the [system-thinking case study](../02_system_thinking_casestudy.md), user feedback functions as a correction layer but is too slow at demo scale to protect product credibility when incorrect questions surface during gameplay.

Key constraints and observations from Phase 1 and early Phase 2 experiments:
- Incorrect or hallucinated questions, if they appear in-game, directly impact user trust.
- Early experiments (≈60 generated questions across two chapters) showed no hallucinations but dataset size is still small and does not eliminate future risk.
- 100% manual verification does not scale and is operationally expensive.
- Runtime user feedback is a weak and delayed signal at demo scale and cannot serve as the primary quality control mechanism.

The system therefore requires an **offline quality gate** that is efficient, human-in-the-loop, and proportional to current scale.

## Options considered

1. **Full manual review**  
   Manually inspect every generated question before publishing.

2. **No active quality control**  
   Rely on prompting, grounding, and runtime user feedback to surface errors.

3. **Sampling-based quality control**  
   Review a statistically meaningful subset of each batch and escalate only if issues are detected.

## Selected option
Option 3 (Sampling-based quality control)

## Rationale

This decision is informed by [system-thinking analysis](../02_system_thinking_casestudy.md) of how errors accumulate over time. Even a small error rate, if unchecked, can compound as synthetic data is reused or fine-tuned against in later phases (e.g. for SLM training).

The `Silver` dataset produced by `qa_validation` pipeline has already passed all automated validation; manual sampling at this stage is intended to detect residual errors that automated checks may miss before promotion to Gold. Sequential acceptance sampling is well-suited here because it assumes that most batches are likely to be acceptable and focuses human effort only when evidence suggests otherwise. It provides a practical and proactive balance between confidence and effort, shifting quality control *left* into the offline Content Factory while avoiding unnecessary manual work.

### Quality control protocol (demo scale)

- For each run, a small, human-manageable random sample of generated questions is reviewed (e.g. ~5–10% of questions adjusted based on run size).
- If no issues are found, the batch is accepted.
- If issues are found, additional questions from the same run are reviewed to assess whether the problem is isolated or systemic.
- If errors appear systemic, the run is rejected and regenerated after the underlying issue is addressed, rather than manually repaired.

Sampling proportions and review depth are intentionally flexible and will be adjusted based on observed quality, run size, and reviewer effort.

## Trade-offs

**Pros:**
- Reduces silent accumulation of incorrect / hallucinated questions.
- Dramatically reduces manual review effort for high-quality batches. It keeps the validation effort proportional to actual risk.  
- Provides a clear, repeatable quality gate during early project phases where user feedback is not yet a reliable signal.
- Aligns with offline-first, batch-oriented design  
- Scales naturally as dataset volume increases  

**Cons:**
- Rare low-error batches may statistically pass sampling despite containing mistakes
- Requires human review when issues are detected  
- Adds operational steps before dataset is ready for the `publisher` pipeline / `Gold` transformation.

## Risks

- Sampling depth and review effort may need adjustment early on until error patterns and failure modes become clearer.
- Rare but systematic errors may evade detection in small samples

**Mitigation:**
- Limit user exposure through short gameplay sessions (~15 questions as per [system-thinking case study](../02_system_thinking_casestudy.md))
- Retain manual correction as a secondary safety net rather than a primary quality control mechanism
- Revisit thresholds once empirical error rates are observed

## Consequences & future considerations

- Quality control becomes an explicit, visible stage in the Content Factory pipeline  
- User feedback remains a secondary safety net rather than the primary control loop  
- As scale increases, additional automated signals (e.g. semantic checks) can augment sampling  
- Automation is intentionally deferred until empirical error rates and generation volume justify additional complexity
