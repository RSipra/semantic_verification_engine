# ADR-P2-018: Stateless Game Architecture for Cloud Deployment

**Status:** 🚧 Proposed<br>
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision
Refactor the trivia game from a stateful CLI loop into a stateless request/response architecture to enable serverless deployment on Cloud Run while preserving the existing rich console experience.

## Options considered
1. **Stateful CLI-style architecture**
   - Maintain a continuous game loop with in-memory state (score, player progress).
   - Attempt to adapt it directly for web serving.

2. **Stateless request/response architecture**
   - Externalize all game state into a serializable session object.
   - Refactor the controller into a pure logic engine.

## Selected option
Option 2 (Stateless request/response architecture)

## Rationale
The Phase 1 MVP used a stateful `while` loop that kept player progress in the VM's active memory. This is incompatible with serverless environments, which forget everything once a request is finished. To resolve this without losing the need to eliminate terminal aesthetic, requires:
- **The bookmark pattern**:  All persistent state is removed from the Controller. Instead, a `GameSession` object containing only plain data (player name, score, progress) is passed between FastAPI and the Controller on each request to resume gameplay.
- **Stateless controller logic**   The `GameController` no longer stores player data in memory. It processes a single turn at a time by accepting a `GameSession`, applying game logic, and returning an updated session..
- **Server-Side Terminal Rendering (SSTR)**: The user browser remains a dumb terminal. The FastAPI server uses the existing `rich` library to generate the visual frame, wraps it in HTML, and ships it to the user. This keeps the logic in Python while providing a modern web interface.

## Trade-offs
**Pros:**
- Enables deployment on Google Cloud Run (serverless)
- Allows scale-to-zero operation with no idle cost
- Allows multiple users to play concurrently, since each request carries its own game state and no player data is stored in shared memory.
- Aligns with stateless serving patterns used in ML/AI systems

**Cons:**
- Increased architectural complexity compared to a linear CLI loop
- Requires explicit phase and session tracking
- Each interaction incurs a full request/response round-trip

## Risks
- Session handling logic may grow complex if not well-bounded
- Increased sensitivity to latency for interactive gameplay

## Consequences & Future considerations
- The application becomes cloud-native and deployment-ready
- The stateless controller pattern mirrors real-world ML inference services
- Future enhancements (semantic checking, SLM integration) can be added without revisiting core runtime architecture
