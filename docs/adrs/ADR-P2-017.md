# ADR-P2-017: Serverless Deployment (FastAPI)

**Status:** ✅ Accepted  
**Phase:** Phase 2 (System & Semantic Intelligence)

## Decision

Transition the runtime from a persistent VM to a stateless, serverless container environment. This will move the system design away from infrastructure-heavy management to a cloud-native architecture with automatic scaling and reduced operational burden.

## Options considered
1. **continue with always-on hosting VM**. Requires manual management of the OS, security patches, and firewalls. Incurs costs even when idle and has a fixed performance ceiling with concurrent users.
2. **Serverless container deployment(Google Cloud Run)**: Provides a managed environment that scales to zero when idle and scales up instantly to handle multiple concurrent users.

## Selected option
Option 2 (serverless container deployment)

## Rationale

While Phase 1 proved the game logic on a VM, Phase 2 aims for a more robust, professional architecture. This choice is driven by three key factors:

- **Scalability and concurrency**: Unlike a single-session CLI on a VM, a FastAPI serverless deployment allows the application to handle multiple concurrent users by spinning up additional instances as needed.
- **Operational decoupling**: Moving to serverless removes the VM management overhead (UFW, OS updates, SSH tunnels). This allows the focus to shift entirely to application logic and semantic intelligence.
- **Cost efficiency**: By utilizing a *scale-to-zero* model, the project remains publicly accessible 24/7 runnig within the Google Cloud Run free-tier limits. The game can scale elastically to meet demand while effectively scaling to zero during idle periods. This move eliminates the $4/month Static IP cost and shifts the focus from managing infrastructure to optimizing application logic and semantic intelligence.

## Trade-offs

**Pros:**
- Zero idle cost. No billing for time when the game is not being played.
- Infrastructure abstraction. No server maintenance, OS patching, or manual firewall (UFW) management.
- Elastic scale. Automatically handles traffic spikes without manual intervention.
- Improved security. Ephemeral containers provide a reduced attack surface compared to a persistent VM.

**Cons:**
- Cold start latency on first request
- Statelessness requirement. The game can not rely on local memory to keep track of players; state must be managed via API requests or external storage.
- Less control over underlying infrastructure

## Risks
- Cold start UX. Latency might make the first interaction feel slow to a new user.
- Resource constraints. Heavy NLP models or long-running logic must be optimized to fit within the serverless memory and timeout limits.

## Consequences & Future considerations
- Architecture shift. The game logic must be refactored into a stateless FastAPI wrapper.
- State management. Scores and session data will be handled via client-side tokens or a lightweight database rather than persistent VM memory.
- Deployment pipeline. The in-situ VM build will be replaced by a modern CI/CD flow where images are pushed to a registry (e.g., Artifact Registry) and deployed automatically.
- AI/SLM integration. Any Small Language Models (SLMs) used for semantic checking must be lightweight enough to load quickly during container startup and execute within request timeouts.
